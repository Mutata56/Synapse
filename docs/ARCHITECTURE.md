# Как устроен Synapse

Заметки тут про внутренности: что писалось руками, где пришлось не брать готовую библиотеку, и какие оптимизации в итоге стоят в коде. Без воды, по подсистемам.

## Стек вкратце

Фронт - React 18 + TypeScript на Vite, состояние на Zustand, стили на Tailwind v4. Редактор - BlockNote (поверх ProseMirror) с кучей своих блоков. Десктоп - Tauri 2, то есть Rust-оболочка вокруг вебвью. Заметки - обычные `.md` файлы на диске, без базы данных.

Дальше - три места, где не получилось обойтись готовым решением: граф связей, интерактивная доска и слой работы с файлами.

## Граф связей: свой физический движок

Граф рисует папки и заметки и тянет между ними рёбра. Раскладку (кто где висит) считает не d3-force и не ForceAtlas2 - они даже выпилены из зависимостей за ненадобностью - а собственный движок в `src/lib/forceLayout.ts`.

На каждый узел на каждом шаге действуют пять сил: кулоновское отталкивание (чтобы узлы не слипались), пружины рёбер, мягкая анти-коллизия дисков, слабая гравитация к центру и когезия кластеров (заметки одной папки сбиваются в остров). Все коэффициенты лежат в одном месте и крутятся слайдерами на лету:

```ts
export const DEFAULT_FORCE_SETTINGS: ForceSettings = {
  repulsion: 20000,
  stiffness: 0.08,
  springLog: true,     // лог-закон Eades вместо линейного Гука: длинные рёбра не "щёлкают"
  cohesion: 0.09,
  interCluster: 10000, // разнос островов-папок
  gravity: 0.025,
  damping: 0.9,        // трение
  collision: 0.7,
  theta: 0.8,          // угол раскрытия Barnes-Hut
  maxSpeed: 80,
  dt: 0.6,
};
```

Интегратор - Verlet по позиции, а не явный Эйлер. Скорость хранится неявно как разница «текущая минус прошлая позиция», а трение это просто домножение этой разницы на `damping`. Verlet стабильнее: граф плавно расслабляется, а не дрожит. Вот само ядро шага:

```ts
const ax = (fx[i] - gravity * (node.x - bx)) / node.mass;
const ay = (fy[i] - gravity * (node.y - by)) / node.mass;
let dx = (node.x - node.px) * damping + ax * dt2;
let dy = (node.y - node.py) * damping + ay * dt2;
const step2 = dx * dx + dy * dy;
if (step2 > maxStep2) {            // зажим скорости, иначе симуляция "взрывается"
  const k = maxStep / Math.sqrt(step2);
  dx *= k; dy *= k;
}
node.px = node.x; node.py = node.y;
node.x += dx;     node.y += dy;
```

### Barnes-Hut вместо перебора всех пар

Отталкивание это единственная сила, которая в лоб требует сравнить каждый узел с каждым - то есть O(n²). На паре сотен заметок это уже заметно. Поэтому пространство делится на quadtree, и если целая ячейка достаточно далеко и мелкая по угловому размеру, она считается одним «псевдозарядом» в своём центре масс, без захода внутрь. Стоимость падает до O(n log n):

```ts
const size = 2 * cell.h;
const hasChildren = cell.c0 >= 0 || cell.c1 >= 0 || cell.c2 >= 0 || cell.c3 >= 0;

// критерий мультиполя: s²/d² < θ². маленькая далёкая ячейка = один заряд
if (!hasChildren || size * size < theta2 * d2) {
  const f = (repulsion * qi * cell.mass) / (dEff * dEff);
  fx -= f * dx * invd;
  fy -= f * dy * invd;
} else {
  if (cell.c0 >= 0) stack[sp++] = cell.c0;   // иначе спускаемся в детей
  if (cell.c1 >= 0) stack[sp++] = cell.c1;
  if (cell.c2 >= 0) stack[sp++] = cell.c2;
  if (cell.c3 >= 0) stack[sp++] = cell.c3;
}
```

Дерево пересобирается каждый шаг (позиции же двигаются), но ячейки берутся из своего пула и обходятся не рекурсией, а ручным стеком на массиве - в установившемся режиме аллокаций ноль, то есть сборщик мусора не дёргается на каждом кадре.

### Откуда берутся узлы и рёбра

Это чистая функция от дерева заметок (`src/lib/buildGraph.ts`), без всякого IPC, так что граф дёшево пересобирается при любом изменении. Рёбра-ссылки достаются из `[[вики-ссылок]]` резолвом по заголовку:

```ts
const titleToKey = new Map<string, string>();
for (const note of flattenNotes(tree)) {
  const t = note.title.trim().toLowerCase();
  if (t && !titleToKey.has(t)) titleToKey.set(t, noteKey(note.id));
}
// ...
for (const target of note.links) {
  const toKey = titleToKey.get(target);
  if (toKey && toKey !== key) graph.mergeEdge(key, toKey, { kind: "link" });
}
```

### Рендер и где он экономит

Рисуется всё на Pixi.js (WebGL). Пара решений, которые держат кадры лёгкими:

- **Симуляция засыпает.** Шаг физики возвращает суммарную кинетическую энергию; как только она падает ниже порога на 40 кадров подряд, цикл просто останавливается. Смотришь на статичный граф - процессор не греется.

```ts
const tick = () => {
  if (!awake) return;
  const energy = focusIdx === null && !frozen && !filterActive ? engine.step() : 0;
  render();
  if (dragIdx === -1 && energy < sleepEnergy) {
    if (++calm > SLEEP_FRAMES) awake = false;   // осело - паркуем
  } else {
    calm = 0;
  }
};
```

- **Кэш прямоугольника канваса.** `getBoundingClientRect()` форсит синхронный layout браузера, а перевод координат курсора в мир зовёт его на каждый `pointermove` - то есть с частотой опроса мыши, вплоть до 1000-8000 Гц. Поэтому rect кэшируется и сбрасывается только на ресайз/скролл:

```ts
let rectCache: DOMRect | null = null;
const invalidateRect = () => { rectCache = null; };
const getRect = (): DOMRect => {
  if (!rectCache) rectCache = canvas.getBoundingClientRect();
  return rectCache;
};
window.addEventListener("resize", invalidateRect);
window.addEventListener("scroll", invalidateRect, true);
```

- **Гало узлов** это одна «запечённая» текстура с радиальным градиентом, на каждый узел вешается её тонированная копия. Никакого покадрового блюра. Все рёбра рисуются одним объектом `Graphics`. Оболочки кластеров - GPU-metaball через свой GLSL-шейдер.

### Цвета берём из CSS, а не хардкодим

Граф на WebGL, но его палитра должна совпадать с темой из `index.css`. Pixi хочет цвет числом `0xRRGGBB`, а CSS даёт что угодно: `#hex`, `rgb()`, именованные, `oklch()`, даже `var(--x)`. Парсить все эти пространства руками - гиблое дело, поэтому цвет резолвит сам браузер: присваиваем выражение скрытому `<span>` и читаем `getComputedStyle`, который всегда отдаёт обычный `rgb()`:

```ts
function resolveToInt(expr: string): number | null {
  const el = probeEl();
  if (el) {
    el.style.color = "";
    el.style.color = expr;            // невалидное значение CSSOM просто отбросит
    if (el.style.color) {
      const n = parseRgb(getComputedStyle(el).color); // всегда rgb(), даже из oklch()
      if (n !== null) return n;
    }
  }
  return parseRgb(expr);
}
```

Смена темы при этом ловится `MutationObserver` и перетинчивает узлы, гало и оболочки. Граф не пересобирается, раскладка и камера остаются на месте.

## Интерактивная доска: свой движок на Canvas 2D

Доска внутри заметки (фигуры, стрелки, текст, рисование от руки) это тоже самописный движок - `src/editor2026/blocks/whiteboard/`, ядро `engine.ts` на чистом TypeScript, около 3400 строк. React тут только тонкая обёртка: всё мутабельное состояние (фигуры, связи, камера, выделение, undo/redo) живёт в движке, мимо React-стейта - иначе перерисовки убивали бы отзывчивость.

Холста два: нижний `world` (сетка, рёбра, спрайты узлов) и верхний `overlay`, который ловит вообще все события указателя. World двигается дешёвым CSS-translate и расширен на буфер в 256px с каждой стороны, чтобы мелкий пан не оголял края.

Рендер устроен на dirty-флаге: кадр рисуется только если что-то поменялось, в простое цикл крутится вхолостую.

```ts
private frame = (): void => {
  if (this.destroyed) return;
  this.drainPendingPointer();    // отложенный pointermove обрабатываем до рендера
  if (this.dirty) {
    this.dirty = false;
    this.render();               // рисуем ТОЛЬКО когда есть что
  }
  this.rafId = requestAnimationFrame(this.frame);
};
```

Самое заметное узкое место было в движениях мыши. Игровые мыши шлют `pointermove` с частотой 1000-8000 Гц, это 8-16 событий на один отрисованный кадр, и каждое тянуло за собой пересчёт. Поэтому почти всё, кроме рисования пером, складывается в «последний event» и потребляется один раз за кадр. Перо - исключение, ему важен каждый сэмпл штриха:

```ts
private onPointerMove = (e: PointerEvent): void => {
  if (this.drag.kind === "draw") {     // перо рисуем синхронно
    this.processPointerMove(e);
    return;
  }
  this.pendingPointerMove = e;         // остальное откладываем до rAF-кадра
  this.markDirty();
};
```

Дальше по мелочи, но всё важное:

- **Кэш спрайтов на фигуру + атлас по визуальному ключу.** Каждая видимая фигура один раз печётся в маленький offscreen-битмап, а пан превращается в обычный `drawImage`. Одинаковые узлы делят один битмап (по визуальному ключу с подсчётом ссылок), плюс LRU-вытеснение, чтобы не разрастаться.
- **SpatialHash** (ячейка 256 единиц): и отрисовка, и попадание по фигуре работают только с видимыми и соседними ячейками, так что стоимость кадра примерно O(видимого), а не O(всех фигур).
- **Рёбра хранят концы как ссылки на узлы, а не координаты** - подвинул узел, путь стрелки пересчитался сам.
- Запись доски в заметку - debounce на 600 мс, чтобы автосейв не громил историю редактора.

Сама доска подключена как кастомный блок BlockNote: в заметке это компактная карточка с превью, а по клику открывается на весь экран через портал.

## Файловая система: всё в обычных файлах

Источник правды - папка `notes/` на диске. Никакой базы (ни SQLite, ни IndexedDB); `localStorage` держит только мелочь вроде языка и ширины сайдбара, но не контент.

Одна заметка это один `.md`: frontmatter сверху плюс тело Markdown. Важная деталь - **id заметки это путь файла**, а не поле в метаданных. Теги и `[[ссылки]]` нигде не хранятся, они вытаскиваются из тела при чтении. Само дерево строится рекурсивным обходом с семафором на 48 параллельных чтений и кэшем по mtime, чтобы не перечитывать то, что не менялось.

Служебное прячется в дотфайлы рядом с заметками: `.trash/`, `.assets/`, `.versions/`, `.settings.json`, `.folder-meta.json`, `.tasks.json`. Поэтому настройки и цвета папок «переезжают» вместе с папкой - скопировал `notes/`, переехало всё.

Картинки дедуплицируются по содержимому: имя файла в `.assets/` это SHA-256 его байтов, так что два одинаковых импорта дают один файл на диске:

```ts
const hash = await sha256Hex(bytes);
const name = `${hash}.${safeExt}`;
const rel = `${ASSETS}/${name}`;
if (!(await pathExists(rel))) {
  await writeFile(rel, bytes, APP_DATA);   // одинаковые байты = один файл
}
return { name, url: await assetUrl(name) };
```

Корзина это не удаление, а перемещение файла в `.trash/` (восстановление - обратный rename). История версий - снапшоты `.versions/<id>/<timestamp>.md` перед перезаписью, с троттлингом раз в 10 минут и лимитом 50 штук.

Бэкап пакует весь воркспейс в `.zip` самописным сборщиком. Самое аккуратное место - восстановление: распакованное складывается рядом, и только потом делается атомарный своп переименованиями. Если второй rename упадёт, обязательно откатываемся, иначе человек останется вообще без папки:

```ts
const hadOld = await safeExists("notes");
if (hadOld) await rename("notes", backupDir, renameOpts);   // старое в notes.bak-<ts>
try {
  await rename(stagingDir, "notes", renameOpts);            // распакованное становится notes
} catch (swapErr) {
  if (hadOld) await rename(backupDir, "notes", renameOpts);  // не вышло - возвращаем как было
  throw swapErr;
}
```

## Rust-бэкенд: только то, что нельзя в вебвью

Rust намеренно тонкий. Файлами он не занимается - вся работа с диском идёт из TypeScript через `@tauri-apps/plugin-fs` со скоупом `$APPDATA`. На Rust вынесено ровно то, чего вебвью не умеет:

- глобальный хоткей быстрой заметки, иконка в трее и безрамочное окно поверх всех; закрытие главного окна сворачивает в трей, чтобы хоткей продолжал ловиться;
- пять команд: переназначить хоткей на лету, скачать `.ics` в обход CORS, и три на CalDAV (найти календари, выгрузить и удалить событие в Яндекс.Календаре).

## Поиск

Постоянного индекса на диске нет. Нечёткий поиск (Fuse.js) стоит только в командной палитре по действиям и заголовкам. Полнотекст по телам - простой `indexOf` с in-memory кэшем, который сам сбрасывается по `updatedAt`, так что перечитываются только изменённые заметки. Подсветка внутри заметки идёт через CSS Custom Highlight API, не вставляя ничего в сам документ.
